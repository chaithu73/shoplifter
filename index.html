Docker Basics — Installation & First Container

Learn to install Docker, run containers, and understand the core image/container model.

Verify Installation
Check Docker is correctly installed and confirm the daemon is running.
# Check Docker version
docker --version

# Verify daemon is running
docker info

# Run the canonical hello-world image
docker run hello-world
Pull & Run Images
Pull images from Docker Hub and run containers in different modes.
# Pull an image without running it
docker pull nginx:latest

# Run interactively in foreground
docker run -it ubuntu:22.04 /bin/bash

# Run detached (background) with name and port mapping
docker run -d --name webserver -p 8080:80 nginx

# List running containers
docker ps

# List ALL containers (including stopped)
docker ps -a
Basic Container Lifecycle
Start, stop, restart, and remove containers.
# Stop a running container
docker stop webserver

# Start a stopped container
docker start webserver

# Restart container
docker restart webserver

# Remove a stopped container
docker rm webserver

# Force remove a running container
docker rm -f webserver

# View container logs
docker logs -f webserver
02

Dockerfile — Building Custom Images

Write Dockerfiles to package your application into a reproducible, layered image.

Basic Dockerfile (Node.js App)
A production-ready Node.js Dockerfile following best practices.
# Use a specific version (never just 'latest' in production)
FROM node:20-alpine

# Set working directory inside container
WORKDIR /app

# Copy package files first (leverages layer caching)
COPY package*.json ./

# Install dependencies
RUN npm ci --only=production

# Copy the rest of the source code
COPY . .

# Expose the port the app listens on
EXPOSE 3000

# Define healthcheck
HEALTHCHECK --interval=30s --timeout=3s \
  CMD wget -qO- http://localhost:3000/health || exit 1

# Run as non-root user for security
USER node

# Default command to run the app
CMD ["node", "server.js"]
Build & Tag Images
Build images from your Dockerfile and tag them for registries.
# Build with a tag
docker build -t myapp:1.0.0 .

# Build with multiple tags
docker build -t myapp:1.0.0 -t myapp:latest .

# Build targeting a specific Dockerfile
docker build -f Dockerfile.prod -t myapp:prod .

# View all local images
docker images

# Tag an existing image for a registry
docker tag myapp:1.0.0 myrepo/myapp:1.0.0

# Push to Docker Hub
docker push myrepo/myapp:1.0.0
Multi-Stage Build
Use multi-stage builds to create lean production images by separating build and runtime environments.
# ─── Stage 1: Build ───────────────────────────
FROM node:20 AS builder
WORKDIR /app
COPY . .
RUN npm ci && npm run build

# ─── Stage 2: Production (tiny image) ─────────
FROM node:20-alpine
WORKDIR /app
# Copy only the built artifacts from the builder
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
USER node
CMD ["node", "dist/index.js"]

Docker Compose — Multi-Container Apps

Orchestrate multiple containers locally with Docker Compose YAML definitions.

docker-compose.yml (Full Stack App)
Define a web app + database + cache stack in one compose file.
version: '3.9'

services:
  app:
    build:
      context: .
      target: development
    ports:
      - "3000:3000"
    environment:
      - DATABASE_URL=postgres://user:pass@db:5432/mydb
      - REDIS_URL=redis://cache:6379
    volumes:
      - .:/app                     # live reload in dev
      - /app/node_modules
    depends_on:
      db:
        condition: service_healthy
      cache:
        condition: service_started

  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_USER: user
      POSTGRES_PASSWORD: pass
      POSTGRES_DB: mydb
    volumes:
      - pg_data:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U user"]
      interval: 10s
      retries: 5

  cache:
    image: redis:7-alpine
    command: redis-server --save "" --appendonly no

volumes:
  pg_data:
Compose Commands
Essential commands for managing your Compose stack.
# Start all services (build if needed)
docker compose up --build

# Start in background
docker compose up -d

# View logs for all services
docker compose logs -f

# Scale a specific service
docker compose up --scale app=3

# Execute command in running container
docker compose exec app sh

# Stop and remove containers + networks
docker compose down

# Also remove volumes (destroys data!)
docker compose down -v

Volumes, Networks & Environment Config

Persist data, isolate services with networks, and manage secrets safely.

Volume Management
Create named volumes and bind mounts for persistent data.
# Create a named volume
docker volume create mydata

# Run with named volume mounted
docker run -v mydata:/app/data myapp

# Bind mount (host path:container path)
docker run -v $(pwd)/data:/app/data myapp

# Read-only bind mount
docker run -v $(pwd)/config:/app/config:ro myapp

# List volumes
docker volume ls

# Inspect a volume
docker volume inspect mydata

# Remove unused volumes
docker volume prune
Network Management
Create custom networks so containers can communicate securely by name.
# Create a custom bridge network
docker network create app-network

# Run container connected to that network
docker run --network app-network --name api myapp

# Connect a running container to a network
docker network connect app-network webserver

# List networks
docker network ls

# Inspect network (see connected containers)
docker network inspect app-network
Environment Variables & .env
Pass configuration to containers without baking secrets into images.
# Pass env vars at runtime
docker run -e NODE_ENV=production -e PORT=3000 myapp

# Load from a .env file
docker run --env-file .env.production myapp

# In docker-compose — reference .env automatically
# .env file in project root is loaded by default
# Override with: --env-file custom.env

# Print env inside a running container
docker exec myapp env

Registry, Cleanup & Production Tips

Push images to registries, clean up resources, and apply production hardening.

Registry Operations
Login, push, and pull from Docker Hub or a private registry.
# Login to Docker Hub
docker login

# Login to a private registry
docker login registry.mycompany.com

# Tag and push to a registry
docker tag myapp:1.2.0 registry.mycompany.com/myapp:1.2.0
docker push registry.mycompany.com/myapp:1.2.0

# Pull from private registry
docker pull registry.mycompany.com/myapp:1.2.0

# Save image as a tar file (air-gapped envs)
docker save -o myapp.tar myapp:1.2.0

# Load image from tar file
docker load -i myapp.tar
Cleanup & System Prune
Reclaim disk space by removing unused images, containers, and volumes.
# Remove dangling images (untagged)
docker image prune

# Remove all unused images
docker image prune -a

# Remove all stopped containers
docker container prune

# NUCLEAR: Remove everything unused
# (images, containers, networks, cache)
docker system prune -a --volumes

# Check disk usage
docker system df

Deployments & ReplicaSets

Declare desired state, roll out updates, and rollback changes safely.

Deployment Manifest
A production-grade Deployment with resource limits and rolling update strategy.
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
  namespace: production
  labels:
    app: myapp
spec:
  replicas: 3
  selector:
    matchLabels:
      app: myapp
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 0
  template:
    metadata:
      labels:
        app: myapp
    spec:
      containers:
      - name: myapp
        image: myrepo/myapp:1.0.0
        ports:
        - containerPort: 3000
        resources:
          requests:
            cpu: "100m"
            memory: "128Mi"
          limits:
            cpu: "500m"
            memory: "512Mi"
        livenessProbe:
          httpGet:
            path: /health
            port: 3000
          initialDelaySeconds: 15
          periodSeconds: 20
Deployment Commands
Apply, update, scale, and rollback Deployments.
# Apply manifest
kubectl apply -f deployment.yaml

# Update image (triggers rolling update)
kubectl set image deployment/myapp \
  myapp=myrepo/myapp:2.0.0

# Watch rollout progress
kubectl rollout status deployment/myapp

# View rollout history
kubectl rollout history deployment/myapp

# Rollback to previous version
kubectl rollout undo deployment/myapp

# Rollback to a specific revision
kubectl rollout undo deployment/myapp --to-revision=2

# Scale manually
kubectl scale deployment/myapp --replicas=5

Services & Ingress — Exposing Applications

Route traffic to pods via Services and expose apps externally via Ingress.

Service Types
Expose pods internally (ClusterIP), externally on node (NodePort), or via cloud load balancer (LoadBalancer).
apiVersion: v1
kind: Service
metadata:
  name: myapp-svc
spec:
  type: ClusterIP           # ClusterIP | NodePort | LoadBalancer
  selector:
    app: myapp             # Matches pod labels
  ports:
  - protocol: TCP
    port: 80              # Cluster-internal port
    targetPort: 3000     # Container port

---
# Quick expose command
kubectl expose deployment myapp \
  --port=80 --target-port=3000 --type=LoadBalancer
Ingress (nginx)
Route HTTP traffic by hostname/path to different services using an Ingress resource.
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: myapp-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
    cert-manager.io/cluster-issuer: "letsencrypt-prod"
spec:
  ingressClassName: nginx
  tls:
  - hosts:
    - myapp.example.com
    secretName: myapp-tls
  rules:
  - host: myapp.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: myapp-svc
            port:
              number: 80
      - path: /api
        pathType: Prefix
        backend:
          service:
            name: api-svc
            port:
              number: 80

ConfigMaps, Secrets & Persistent Volumes

Externalize configuration, handle sensitive data, and persist storage.

ConfigMap
Store non-sensitive configuration as key-value pairs or mounted files.
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  NODE_ENV: "production"
  LOG_LEVEL: "info"
  config.json: |
    {
      "feature_flags": { "dark_mode": true },
      "max_connections": 100
    }

---
# Use in a Pod spec:
envFrom:
- configMapRef:
    name: app-config

# Or mount as a file volume:
volumes:
- name: config-vol
  configMap:
    name: app-config
Secrets
Store passwords, tokens, and keys. Values must be base64-encoded.
# Create secret imperatively (safest for terminals)
kubectl create secret generic db-creds \
  --from-literal=DB_PASS=s3cr3t \
  --from-literal=DB_USER=admin

# Create from file
kubectl create secret generic tls-certs \
  --from-file=tls.crt --from-file=tls.key

# YAML form (values are base64 encoded)
apiVersion: v1
kind: Secret
type: Opaque
metadata:
  name: db-creds
data:
  DB_USER: YWRtaW4=       # echo -n "admin" | base64
  DB_PASS: czNjcjN0       # echo -n "s3cr3t" | base64
PersistentVolume & PVC
Request durable storage that survives pod restarts using PersistentVolumeClaims.
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: postgres-pvc
spec:
  accessModes:
  - ReadWriteOnce
  storageClassName: standard
  resources:
    requests:
      storage: 10Gi

---
# Mount in pod spec:
volumes:
- name: pg-storage
  persistentVolumeClaim:
    claimName: postgres-pvc
volumeMounts:
- mountPath: /var/lib/postgresql/data
  name: pg-storage

HPA, RBAC & Namespaces — Production Ops

Autoscale workloads, enforce access control, and isolate environments with namespaces.

Horizontal Pod Autoscaler
Automatically scale pod replicas based on CPU/memory metrics.
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: myapp-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: myapp
  minReplicas: 2
  maxReplicas: 20
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 60
  - type: Resource
    resource:
      name: memory
      target:
        type: Utilization
        averageUtilization: 70

# Quick command form
kubectl autoscale deployment myapp \
  --cpu-percent=60 --min=2 --max=20
RBAC — Role-Based Access Control
Define fine-grained permissions using Roles and RoleBindings.
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: pod-reader
  namespace: production
rules:
- apiGroups: [""]
  resources: ["pods", "pods/log"]
  verbs: ["get", "list", "watch"]

---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: read-pods-binding
  namespace: production
subjects:
- kind: User
  name: jane@company.com
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: Role
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io
Namespaces & Context
Organize resources by team or environment with namespaces.
# Create namespace
kubectl create namespace staging

# Apply resource to a specific namespace
kubectl apply -f deployment.yaml -n staging

# Set default namespace for current context
kubectl config set-context --current \
  --namespace=production

# View resource quota
kubectl get resourcequota -n production

# Set resource quota on a namespace
kubectl create quota dev-quota \
  --hard=cpu=4,memory=8Gi,pods=20 \
  -n development
